In this blog post we going to analyze the *Linux x86 chmod Payload* . In order to undestand what this payload does we going to debug the program with GDB, LIBEMU, NISAM. This kind of analysis is very important when you need to identify the underlying functionality of a unknown shellcode.


### Linux x86 chmod Payload

This payload use the chmod syscall to change the permissions of the file specified in the FILE options. Using the `msfvenon`tool we can interact with this payload.


![alt text][img-1]

[img-1]: https://h4pp1n3ss.github.io/assets/images/p51-1.png "MSFVenon payload options"



In the next image we use the `-f elf > filename` argument to create the ELF file. We also change the permission of the file and with the `file` command we can see the format of the file. Also with the `hd` command we can see the `ELF HEADER` of the file and other values.

![alt text][img-2]

[img-2]: https://h4pp1n3ss.github.io/assets/images/p51-2.png "ELF Header"

## NR_chmod syscall

This syscall change the file permission of a file. According to the documentation this sycall need two arguments char `pathname` and `mode_t mode`. In the next Image we can see the documentation from the man.


![alt text][img-3]

[img-3]: https://h4pp1n3ss.github.io/assets/images/p51-3.png "chmod user manual"

We open the file with `gdb` using the `-q` option to skip the banner information.

With the command `readelf -h` we can identify the entry point of the application. According to the output the entry point is located in the address `0x8048054` so we set a break point in this memory address.

![alt text][img-4]

[img-4]: https://h4pp1n3ss.github.io/assets/images/p51-4.png "readelf output"


We set the breakpoint with the command break `0x8048054	` and run the program. In the next Image we hit the break point and see the starting point of the program.


![alt text][img-5]

[img-5]: https://h4pp1n3ss.github.io/assets/images/p51-5.png "Breakpoint"


To get a better idea of what syscalls are used by the program we run the `strace` command. 

![alt text][img-6]

[img-6]: https://h4pp1n3ss.github.io/assets/images/p51-6.png "strace command"

Continuing with the GDB in the next image in address 0x8048071 we can see the int 0x80 instruction (the syscall interrupt) and the values of the registers.

1. **eax** register contains the value `0xf` the number 15 in decimal which is the id of **chmod** syscall.
2. **ebx** register contains the first argument char `*pathname` of the chmod syscall
3. **ecx** register contains the second argument `mode_t` mode of the chmod syscall. This value represent the permission to set.

![alt text][img-7]

[img-7]: https://h4pp1n3ss.github.io/assets/images/p51-7.png "General purpose registry content"


With the command `ls -la /etc/shadow`  we can see the current permission of the file `/etc/shadow`. After the execution of the syscall we can see the permission of the file  was change to 666.


![alt text][img-9]

[img-9]: https://h4pp1n3ss.github.io/assets/images/p51-9.png "File permission changed"
